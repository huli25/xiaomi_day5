## 2. 加权决策矩阵表

### 2.1 权重设定依据

| 因素 | 权重 | 设定理由 |
| --- | --- | --- |
| **安全可控性** | 25% | 题眼核心：越权零容忍、LLM无执行权 |
| **技术可行性** | 25% | 必须满足§0题面门槛 |
| **开发效率** | 20% | 原型交付时间约束 |
| **维护成本** | 15% | 后续迭代与交接 |
| **扩展性** | 15% | AI能力预留空间 |

### 2.2 加权评分矩阵

| 评估因素 | 权重 | 方案一 | 方案二 | 方案三 | 方案四 |
| --- | ---: | ---: | ---: | ---: | ---: |
| **技术可行性** | 25% | 5 (25.0) | 5 (25.0) | 4 (20.0) | 5 (25.0) |
| **安全可控性** | 25% | 5 (25.0) | 5 (25.0) | 3 (15.0) | 5 (25.0) |
| **开发效率** | 20% | 4 (20.0) | 5 (20.0) | 3 (15.0) | 3 (15.0) |
| **维护成本** | 15% | 4 (15.0) | 4 (15.0) | 2 (7.5) | 3 (11.25) |
| **扩展性** | 15% | 2 (7.5) | 2 (7.5) | 5 (18.75) | 4 (15.0) |
| **加权总分** | 100% | **92.5** | **92.5** | **76.25** | **91.25** |

### 2.3 决策结论

| 排名 | 方案 | 总分 | 推荐度 |
| --- | --- | ---: | :---: |
| 🥇 1 | **方案一**（React+Express规则模板） | 92.5 | ⭐⭐⭐⭐⭐ |
| 🥇 1 | **方案二**（Next.js全栈规则模板） | 92.5 | ⭐⭐⭐⭐⭐ |
| 🥉 3 | **方案四**（Vue+FastAPI规则模板） | 91.25 | ⭐⭐⭐⭐ |
| 4 | **方案三**（React+Express模板+LLM） | 76.25 | ⭐⭐⭐ |

> **综合推荐**：方案一或方案二任选其一，均可满足安全可控性满分要求。
> - 选方案一：团队熟悉Express，追求与PRD示例一致的React技术栈
> - 选方案二：追求开发效率最大化，接受Next.js框架约束

## 6. AI/人工分工表

### 6.1 职责边界定义（基于PRD Goals）

| 阶段 | AI职责 | 人工/系统职责 |
| --- | --- | --- |
| **输入解析** | 识别意图关键词、提取候选参数 | 接收文本输入 |
| **任务拆解** | 生成结构化任务列表（对象/参数/顺序） | 提供规则模板定义 |
| **模糊检测** | 识别参数模糊性（开度、程度等） | 定义模糊判据规则 |
| **参数补全** | ❌ **禁止**：AI不得填入"看起来精确"的参数 | 等待用户确认/补参 |
| **安全判定** | ❌ **禁止**：AI不得给出最终decision | 服务端规则引擎是唯一决策源 |
| **风险评估** | 提供风险相关信息 | 执行风险等级判定（execute/confirm/reject） |
| **确认交互** | 生成确认话术 | 展示确认界面、接收用户响应 |
| **执行记录** | 提供执行状态反馈 | 记录日志、更新车态 |

### 6.2 AI能力使用边界（方案三专用）

| 场景 | AI可用 | AI禁用 | 护栏措施 |
| --- | --- | --- | --- |
| 意图识别 | ✅ | - | - |
| 参数提取 | ✅ | - | - |
| 模糊标记 | ✅ | - | - |
| 参数补全 | - | ❌ 填入精确值 | sanitize层强制模糊标记 |
| 决策判定 | - | ❌ 给出execute/confirm/reject | 丢弃AI决策，使用规则引擎结果 |
| 话术生成 | ✅ | - | 确认话术模板 |
| 日志记录 | - | ❌ 作为安全凭证 | 仅记录调用，不作为判定依据 |

```
│         ▼                ▼                ▼                                 │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐                             │
│   │ 人工：无需 │    │ AI/人工：│    │ 人工：展示 │                             │
│   │ 介入       │    │ 等待确认 │    │ 拒绝原因   │                             │
│   │ 直接执行   │    │ 参数补全 │    │ 无执行     │                             │
│   └──────────┘    └──────────┘    └──────────┘                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 6.4 各方案AI定位对照

| 方案 | AI定位 | 说明 |
| --- | --- | --- |
| 方案一 | ❌ 不使用AI | 纯规则模板拆解 |
| 方案二 | ❌ 不使用AI | 纯规则模板拆解（同方案一） |
| 方案三 | ⚠️ 受限使用 | 仅用于拆解辅助，decision由规则引擎决定 |
| 方案四 | ❌ 不使用AI | Python规则引擎拆解 |

### 6.5 安全关键裁定落地检查

| 裁定项 | 落地要求 | 方案一 | 方案二 | 方案三 | 方案四 |
| --- | --- | ---: | ---: | ---: | ---: |
| AI无最终执行权 | decision必须来自规则引擎 | ✅ | ✅ | ✅* | ✅ |
| 模糊不得直执 | 模糊参数必须confirm | ✅ | ✅ | ✅ | ✅ |
| 显式role鉴权 | 忽略话术身份声明 | ✅ | ✅ | ✅ | ✅ |
| 执行前再查车况 | 确认前重新校验车态 | ✅ | ✅ | ✅ | ✅ |

> *方案三：即使LLM给出了某种"建议"，最终decision必须由规则引擎重新判定，丢弃LLM输出中的decision字段。

## 7. 最终建议

### 7.1 推荐方案

| 推荐 | 方案 | 理由 |
| --- | --- | --- |
| **首选** | 方案一<br>React+Express+规则模板 | 安全可控性满分，开发效率高，TypeScript统一栈，维护成本低 |
| **备选** | 方案二<br>Next.js全栈+规则模板 | 同等安全级别，开发体验最佳，适合追求效率的团队 |
| **可选** | 方案四<br>Vue+FastAPI+规则模板 | Python生态强大，适合有Python背景的团队 |
| **不推荐** | 方案三<br>React+Express+模板+LLM | 安全可控性降低，依赖网络，维护复杂度过高 |

### 7.2 决策条件

如满足以下任一条件，可考虑备选方案：

| 条件 | 考虑方案 | 说明 |
| --- | --- | --- |
| 团队熟悉Next.js | 方案二 | 开发体验最优 |
| 团队Python能力强 | 方案四 | Pydantic验证强大 |
| 需要AI增强能力 | 方案三 | 仅在安全可控性可接受的情况下 |
