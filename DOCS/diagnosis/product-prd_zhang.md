## 1. 产品一句话

做一个**可演示的薄原型**：自然语言车控指令经拆解与安全判定后，只能走 **执行 / 需确认 / 拒绝** 三态之一；风险步必须人确认；全过程可追溯。证明重点是 **权限、安全、确认、状态变化、部分失败**，不是功能数量。

## 2. User / 场景

### 2.1 用户

| 用户 | 诉求 | MVP 是否服务 |
| --- | --- | --- |
| **车主（驾驶员）** | 用自然语言完成有限车控；模糊时被问清楚；危险/无权限时被拦住 | 是 |
| **临时访客** | 可表达意图，但无授权操作必须被拒绝且原因可见 | 是（用身份切换演示） |
| **调试/演示人** | 可改车态与身份，构造一正一反与边界样例 | 是（调试能力） |
| 量产乘客 / 远程车队运营 | — | **否（Non-goal）** |

### 2.2 核心场景（必须可演示）

| ID | 场景 | 期望可观察结果 |
| --- | --- | --- |
| S1 | 车主发一条可直执的安全指令（如打开空调到合适范围且参数明确） | 拆解可见 → 至少一步 `execute` → 模拟成功 → 时间线完整 |
| S2 | 车主：「主驾留条缝」（模糊开度） | 拆解标出模糊点 → **不得**直接当精确执行 → `confirm` 或补参后才可继续 |
| S3 | 身份=临时访客，发需授权车控（如解锁/授权类意图） | 意图拆解正确 → 步骤 `reject` → 原因可见 → 无成功执行记录 |
| S4 | 确认后、执行前车态已变（调试页改车速/挡位等）使原判定失效 | 再查后该步 `blocked`，日志可追溯 |
| S5 | 一条指令拆成多步，其中一步失败 | 失败步有记录；**依赖后续步**不得标成功（见 §5 部分失败勾选） |

S2、S3 = 题面线索；S4、S5 及 §8 自补样例 = 隐蔽失败/边界。

## 3. Goals / Non-goals

### 3.1 Goals

1. 跑通题面六步闭环：输入 → 拆解 → 安全检查 → 补充/确认 → 执行/再查 → 最终记录。
2. 每步判定只有三态：`execute` | `confirm` | `reject`（执行结果另有 `success` | `blocked` | `failed`）。
3. **AI 拆解无最终执行权**（继承 Flow-map B1）：有模糊/缺参不得因 AI 填了「看起来精确」的数而直执。  
   → **【团队已确认 · 安全关键裁定 · 2026-07-16】** 后续 Design/实现不得削弱。
4. 演示必须 **一正一反**（存在直执成功与直拒），禁止「全部待确认」。
5. 前端可展示：拆解、判定、确认、逐步结果、终态车态时间线。
6. **提供调试手段**：切换身份（车主/访客）与车态字段，以便触发规则。  
   → **【团队已确认 · 安全关键裁定 · 2026-07-16】** 调试是为了**可验证地触发权限/安全联锁**，不是为了绕过规则；调试不得伪造已成功执行的假日志（Flow-map 角色表）。

### 3.1.1 安全关键裁定说明（已确认）

| 裁定 | 安全含义 | 验收锚点 |
| --- | --- | --- |
| AI 无最终执行权 | 拆解≠授权；防「假精确」直执 | R3、B1、decision 不得由 AI 单独给出 execute 准出 |
| 必须有调试手段 | 安全规则要能被**主动构造**出来验证（身份/车态），否则权限与联锁无法演示 | R4、R6、R9、R10；调试改快照后下一次编排必须吃到新状态 |
### 3.2 Non-goals（主动收窄，不丢分）

| 不做 | 原因 |
| --- | --- |
| 真实麦克风 ASR / 车载语音唤醒 | 两天原型；文本输入即可 |
| 真实车总线 / 实车执行 | 仅 **模拟执行** |
| **真实用户鉴权**（账号登录、车钥匙/蓝牙钥匙绑定、声纹、人脸、车机账号 SSO） | MVP 用前端显式 `role` 切换证明权限规则；见 §4.1；量产再替换鉴权源 |
| 覆盖全部车控功能域 | MVP 只保留能证明判断的最少对象（见 §4）；与「拆解器怎么实现」无关 |
| 多前端形态（小程序+App+车机同时） | 题面：一端前端即可 |
| 多租户账号体系 / 云端车队 | 与题眼无关 |
| 把「语义完整度评分」当验收指标 | 验收只看可观察行为与日志（R↔A），不把准确率分数本身当准出 |

### 3.3 留给 Design 的开放取舍（PRD 不预先收窄）

以下 **不在 Non-goals 里拍死**，须在第 3 棒 Design 作为独立方案（或方案维度）对比优劣后再取舍：

| 开放项 | Design 须回答 | PRD 硬约束（取舍时仍不可破） |
| --- | --- | --- |
| **拆解器路线**：规则/模板 vs **追求 LLM 拆解准确率 / 更广意图覆盖** vs 混合 | 各路线对两天交付、演示稳定性、成本、B1 安全性的影响；给出取舍矩阵与拍板理由 | 无论选哪条：**AI/拆解结果无最终执行权**；模糊/缺参不得直执；P0 场景（含题面两条线索）必须可稳定复现演示 |
| 是否把「准确率/覆盖面」做成可度量目标 | 若采纳 LLM 路线，准确率是研发内部指标还是对外验收？如何与 R↔A 对齐 | 对外准出仍以 R1–R10 可观察行为为准，不以准确率榜单代替 |

**产品意图：** 先把「要不要上 LLM、要不要追准确率与全覆盖」放进 Design 辩论；PRD 只锁安全与可演示验收，避免过早把技术路线写成 Non-goal。

## 4. MVP 能力范围（功能做薄）

### 4.1 身份与权限（如何知道是车主还是访客）

#### 4.1.1 身份取值（恰好 2）

| `role` | 含义 |
| --- | --- |
| `owner` | 车主（驾驶员） |
| `guest` | 临时访客 |

#### 4.1.2 MVP 身份如何确认（产品裁定）

**本原型不实现真实鉴权**（无账号密码、无车钥匙绑定、无声纹/人脸、无车机登录态对接）。

身份来源锁定为：

1. **前端显式选择（权威输入）**  
   主界面或调试区提供「当前身份」切换：`车主` / `临时访客`。  
   用户每发起一次编排，前端把当前选中的 `role` **原样随请求提交**给编排服务。
2. **后端只信请求里的 `role`，不自行猜测**  
   规则引擎用该字段做权限判定；**不得**根据语音内容、座位、或 AI 拆解结果推断身份。
3. **缺省与非法**  
   未传 / 非枚举值 → 按 I4：**不创建会话**，提示合法取值。不默认当成车主（避免静默提权）。
4. **确认等待中的身份变更**  
   若会话处于 `awaiting_confirm`，用户改了身份再点确认：执行前 **再查**须使用**确认当下**的 `role`（或 Design 规定「确认时必须重传 role」）——用于 H2 样例；不得沿用过期身份放行。

**诚实边界（答辩可讲）：**  
MVP 的「身份」是**可切换的权限夹具**，用来证明规则会按身份拦人；量产应替换为真实鉴权签发的身份断言。本 PRD 把真实鉴权列为 Non-goals，避免两天内假造登录体系。

#### 4.1.3 权限如何据此判定（最小权限表）

规则引擎在拆解之后、执行之前，按 `role + 任务对象/动作` 查表（Design 可细化动作枚举，不得削弱下表精神）：

| 对象 / 动作类型 | `owner` | `guest` |
| --- | --- | --- |
| 车窗调节（含模糊开度） | 允许进入后续安全/模糊判定（可为 execute 或 confirm） | 允许进入后续判定（访客通常可调舒适项；若规则收紧可改为 reject——**MVP 默认：允许调窗/空调，禁止授权类**） |
| 空调开关/设温 | 同上 | 同上 |
| 门锁解锁、授权他人、共享钥匙类 | `confirm` 或 `execute`（由安全规则定） | **必须 `reject`**（题面线索②；意图可拆对，权限不过） |
| 未在表内的对象 | `reject`（未知对象） | `reject` |

**判定顺序（产品语义）：**  
身份非法 → 拒会话；身份合法 → 拆解 → **先权限**（本表）→ 再安全车态 → 再模糊/风险确认 → 执行与再查。  
权限 `reject` 时：**不必**再要求用户确认「是否执行」（避免把无权限做成假确认）。

#### 4.1.4 可观察验收（身份相关）

| 操作 | 期望 |
| --- | --- |
| UI 显示当前身份为车主或访客，且可切换 | 切换后下一次提交的编排带上新 `role` |
| 同一句「请授权临时用户开车门」：身份=owner vs guest | guest → 对应步 `reject` 且原因含权限；owner → 不因「是访客」被拒（仍可因其他安全规则 confirm/reject） |
| 时间线 | 记录本次会话使用的 `role`，可追溯 |

### 4.2 车态字段（调试可改；模拟执行可写回）

| 字段 | 含义 | MVP 用途 |
| --- | --- | --- |
| `speedKmh` | 车速 | 行驶中限制部分操作 |
| `gear` | 挡位（如 P/R/N/D） | 安全联锁 |
| `batteryPct` | 电量 0–100 | 可选低电量拒绝高耗操作（若纳入规则） |
| `fault` | 有无故障布尔或枚举 | 故障时拒绝相关执行 |

### 4.3 车控对象（MVP 最少集，可证明即可）

必须能支撑 S1–S5 与题面线索；建议锁定以下对象（Design 可微调命名，不得扩大到「功能大全」）：

| 对象 | 示例意图 | 典型判定 |
| --- | --- | --- |
| 车窗（主驾） | 「留条缝」「开一半」 | 模糊 → confirm/补参 |
| 空调 | 明确温度/开关 | 参数全则可 execute |
| 门锁/授权类 | 访客试图授权或解锁 | guest → reject |

**放弃（写入 options 时可再列）：** 座椅记忆、导航改道、AD 自动驾驶相关、支付、多车联动等。

## 5. 输入边界勾选表（必做 · 已裁定）

> 实现与测试必须按本表；若要改，先改本 PRD 再改 Design/代码。

| # | 边界项 | 裁定（勾选结果） | 空/非法时的可观察响应 |
| --- | --- | --- | --- |
| I1 | 单次指令文本长度 | **1～200 字**（Unicode 字符） | 空：拒绝创建会话，提示「请输入指令」；超长：拒绝，提示超限 |
| I2 | 单次会话拆解任务数上限 | **1～8 步**；超出则整单 `reject` 或截断策略二选一 → **锁定：超出则整单拒绝并说明** | 前端展示拒绝原因，无部分成功 |
| I3 | 并发会话 | **MVP 单用户单活跃会话**；新开可中止旧会话或禁止第二活跃 → **锁定：同一客户端仅 1 个 `awaiting_confirm` 会话** | 重复提交：提示先处理当前确认或取消 |
| I4 | 身份字段 | 仅 `owner` \| `guest`；**必须由前端显式选择并随请求提交**；服务端不推断；缺省按非法 | 非法/缺失：不创建会话，提示合法取值；**禁止默认 owner** |
| I4a | 身份与权限 | 按 §4.1.3 权限表；权限 reject 不再走「假确认」 | 访客授权类意图：可见 reject 原因；时间线含 role |
| I5 | 车态字段范围 | `speedKmh`：0–200；`gear`∈{P,R,N,D}；`batteryPct`：0–100；`fault`：bool | 越界：拒绝本次编排或拒绝调试写入，并提示字段错误（不得静默夹紧后假装合法——**锁定：拒绝并提示**） |
| I6 | 模糊参数 | 开度/比例等未给出可执行精确值 → 不得 `execute` | 必须 `confirm` 或返回「需补充参数」；用户可提交补充值后再判 |
| I7 | 部分失败策略（Flow-map B4） | **锁定：依赖后续步 blocked；无依赖声明时默认「顺序依赖」——失败步之后全部 blocked** | 时间线中失败步=`failed`/`blocked`，后续依赖步=`blocked`，不得标 `success` |
| I8 | 多对象一句话 | 允许拆成多步，但仍受 I2 上限 | 超限整单拒；未超限逐步判定 |
| I9 | 确认超时 | **MVP 不设服务端超时自动执行**；会话可保持等待直到确认/取消 | 无自动 execute |
| I10 | 「文件名」类字段 | **不适用**（本产品无上传包）。原因：输入为文本指令与车态 JSON，无文件名验收项。 | — |

## 6. 需求 ↔ 验收（R↔A）

优先级：P0 不过则红线/Demo 失败；P1 答辩加分与稳健；P2 有时间再做。

| ID | 优先级 | 需求 R | 可观察验收 A |
| --- | --- | --- | --- |
| R1 | P0 | 用户可提交自然语言指令并看到结构化拆解（对象/参数/顺序/模糊点） | 提交合法指令后 UI 出现 tasks 列表；模糊指令可见 ambiguity 标记 |
| R2 | P0 | 每步给出 execute/confirm/reject 之一及原因 | 每步有 decision + reason 文案；三态在同一次 Demo 中至少两类出现（含 reject 或 execute） |
| R3 | P0 | 模糊开度不得直执（题面线索① / B1） | S2：无用户确认/补参前，该步不得出现 success |
| R4 | P0 | 访客无权限必须拒绝（题面线索②）；身份来源为显式 `role`（§4.1） | S3：UI 置 guest 后发授权类指令 → reject；无 success；日志含 role=guest |

| R5 | P0 | 风险/模糊步需人确认或补参后方可继续 | confirm 态下必须有确认/取消；取消后不得 success |
| R6 | P0 | 执行前再查车态（B3） | S4：改车态后原可执行步变为 blocked 或重新 confirm/reject，且日志含再查记录 |
| R7 | P0 | 部分失败按 I7 处理（B4） | S5：失败后后续步均为 blocked，时间线一致 |
| R8 | P0 | 会话可追溯（B5） | 可展开：原文、拆解、逐步判定、确认动作、执行结果、终态车态、**role** |
| R9 | P0 | 调试/主界面可切换身份与车态 | 切换后下一次编排使用新 role/车态快照；确认再查使用确认时身份 |
| R10 | P0 | 一正一反可演示 | 存在脚本/用例：至少一次直执成功 + 一次直拒 |
| R11 | P1 | 自补 ≥3 隐蔽失败样例并写入验证文档 | `DOCS/validation/` 有计划与结果；样例见 §8 |
| R12 | P1 | 空/非法输入符合 §5 | 对 I1/I4/I5 各至少 1 条拒绝用例 |
| R13 | P2 | 截图/录屏兜底进 PROTOTYPE | 主路径与一反路径有静态证据 |
| R14 | P2 | 低电量/故障等扩展规则 | 至少 1 条规则可演示即可，不扩功能面 |

## 7. 判定与结果状态机（产品语义）

### 7.1 步骤判定（规则引擎输出）

| decision | 含义 | 人是否必须动手 |
| --- | --- | --- |
| `execute` | 当前车态+权限下允许模拟执行 | 否 |
| `confirm` | 允许但有风险或参数不清，须人确认/补参 | 是 |
| `reject` | 权限或安全不允许 | 否（仅展示原因） |

### 7.2 步骤执行结果

| result | 含义 |
| --- | --- |
| `success` | 模拟执行成功并写回相关车态 |
| `blocked` | 再查未通过，或因部分失败策略未执行 |
| `failed` | 模拟执行尝试失败（如对象故障） |
| `cancelled` | 用户取消确认 |

**产品禁令：** 不得用「全部 confirm」代替 reject/execute；AI 建议参数≠已确认参数。

## 9. 体验约束（给 Design，非技术选型）

1. 主路径应在同一前端主流程内完成，无需跳转多系统。
2. 判定三态在视觉上可区分（具体视觉语言由 Design 定）。
3. 确认交互必须展示：将执行什么、为何需确认、确认/取消后果。
4. 时间线按时间或步骤序完整展示，答辩时可指认 B1。
5. 调试能力可与主控台同页分区或同应用另一路由，但须同原型可访问。
