### 1.3 用户痛点

- **多目标复杂性**：用户一句复合指令可能包含多个控制目标（如"打开车窗、关空调、调低座椅加热"）
- **参数模糊性**：用户口语表达天然存在模糊性（如"留条缝"、"凉一点"、"稍微高点"）
- **动态车况**：车辆状态（车速、挡位、电量、故障码）实时变化，安全边界随之动态调整
- **权限管控**：不同角色需要不同的操作权限，越权操作必须拦截
- **行车安全**：行驶状态下某些高风险操作必须强制确认或禁止

### 2.4 产品 Goals（来自 product-prd_zhang.md）

#### 2.4.1 六步闭环与三态判定

1. **跑通题面六步闭环**：输入 → 拆解 → 安全检查 → 补充/确认 → 执行/再查 → 最终记录
2. **每步判定只有三态**：`execute` | `confirm` | `reject`（执行结果另有 `success` | `blocked` | `failed`）
3. **AI 拆解无最终执行权**：有模糊/缺参不得因 AI 填了「看起来精确」的数而直执  
   > **【团队已确认 · 安全关键裁定 · 2026-07-16】** 后续 Design/实现不得削弱
4. **演示必须一正一反**：存在直执成功与直拒，禁止「全部待确认」
5. **前端可展示**：拆解、判定、确认、逐步结果、终态车态时间线
6. **提供调试手段**：切换身份（车主/访客）与车态字段，以便触发规则  
   > **【团队已确认 · 安全关键裁定 · 2026-07-16】** 调试是为了可验证地触发权限/安全联锁，不是为了绕过规则

#### 2.4.2 安全关键裁定说明（已确认）

| 裁定 | 安全含义 | 验收锚点 |
| --- | --- | --- |
| AI 无最终执行权 | 拆解≠授权；防「假精确」直执 | R3、B1、decision 不得由 AI 单独给出 execute 准出 |
| 必须有调试手段 | 安全规则要能被主动构造出来验证（身份/车态），否则权限与联锁无法演示 | R4、R6、R9、R10；调试改快照后下一次编排必须吃到新状态 |

## 3. 非目标

以下功能不在本项目范围内：

| 编号 | 非目标 | 说明 |
|-----|-------|------|
| N1 | 不做语音识别 | 假设语音转文本已完成，系统接收文本输入 |
| N2 | 不做自然语言生成 | 确认话术和反馈为预定义模板 |
| N3 | 不做车辆硬件控制 | 只输出控制指令，不直接操作硬件 |
| N4 | 不做机器学习模型训练 | 使用规则引擎，不涉及模型训练 |
| N5 | 不做远程OTA升级 | 规则配置为本地文件，不涉及云端同步 |
| N6 | 不做多车协同 | 单车单系统，不涉及车与车通信 |
| N7 | 不做真实麦克风 ASR / 车载语音唤醒 | 两天原型；文本输入即可 |
| N8 | 不做真实车总线 / 实车执行 | 仅模拟执行 |
| N9 | 不做真实用户鉴权（账号登录、车钥匙/蓝牙钥匙绑定、声纹、人脸、车机账号SSO） | MVP用前端显式role切换证明权限规则；量产再替换鉴权源 |

| N11 | 不做多前端形态（小程序+App+车机同时） | 题面：一端前端即可 |
| N12 | 不做多租户账号体系 / 云端车队 | 与题眼无关 |
| N13 | 不做把「语义完整度评分」当验收指标 | 验收只看可观察行为与日志，不把准确率分数本身当准出 |

## 4. 每条需求验收（R↔A）

> 验收只看可观察行为与日志，不把准确率分数本身当准出（N13）

### 4.1 R1-R10 可观察行为验收（核心）

| ID | 优先级 | 需求 R | 可观察验收 A |
| --- | --- | --- | --- |
| R1 | P0 | 用户可提交自然语言指令并看到结构化拆解（对象/参数/顺序/模糊点） | 提交合法指令后UI出现tasks列表；模糊指令可见ambiguity标记 |
| R2 | P0 | 每步给出 execute/confirm/reject 之一及原因 | 每步有 decision + reason 文案；三态在同一次 Demo 中至少两类出现（含 reject 或 execute） |
| R3 | P0 | 模糊开度不得直执（AI拆解无最终执行权） | 模糊指令无用户确认/补参前，该步不得出现 success |
| R4 | P0 | 访客无权限必须拒绝；身份来源为显式 role | UI置 guest 后发授权类指令 → reject；无 success；日志含 role=guest |
| R4a | P0 | 系统不得从话术内容推断身份 | 话术含「我是车主」但UI选guest → 仍按 guest 鉴权 |
| R5 | P0 | 风险/模糊步需人确认或补参后方可继续 | confirm 态下必须有确认/取消；取消后不得 success |
| R6 | P0 | 执行前再查车态 | 改车态后原可执行步变为 blocked 或重新 confirm/reject，且日志含再查记录 |
| R7 | P0 | 部分失败按顺序依赖处理 | 失败后后续步均为 blocked，时间线一致 |
| R8 | P0 | 会话可追溯 | 可展开：原文、拆解、逐步判定、确认动作、执行结果、终态车态、role |
| R9 | P0 | 调试/主界面可切换身份与车态 | 切换后下一次编排使用新 role/车态快照；确认再查使用确认时身份 |
| R10 | P0 | 一正一反可演示 | 存在脚本/用例：至少一次直执成功 + 一次直拒 |

### 4.2 输入边界勾选表（必做）

| # | 边界项 | 裁定 | 空/非法时的可观察响应 |
| --- | --- | --- | --- |
| I1 | 单次指令文本长度 | **1～200字**（Unicode） | 空：拒绝创建会话；超长：拒绝并提示超限 |
| I2 | 单次会话拆解任务数上限 | **1～8步**；超出则整单 reject | 前端展示拒绝原因，无部分成功 |
| I3 | 并发会话 | **MVP单用户单活跃会话** | 重复提交：提示先处理当前确认或取消 |
| I4 | 身份字段 | 仅 `owner` \| `guest`；必须由前端显式选择 | 非法/缺失：不创建会话，提示合法取值；禁止默认 owner |
| I5 | 车态字段范围 | speedKmh：0-200；gear∈{P,R,N,D}；batteryPct：0-100；fault：bool | 越界：拒绝本次编排或拒绝调试写入，提示字段错误 |
| I6 | 模糊参数 | 开度/比例等未给出可执行精确值 → 不得 execute | 必须 confirm 或返回「需补充参数」 |
| I7 | 部分失败策略 | **锁定：依赖后续步 blocked；默认「顺序依赖」** | 失败步=failed/blocked，后续依赖步=blocked，不得标 success |
| I8 | 多对象一句话 | 允许拆成多步，但仍受 I2 上限 | 超限整单拒；未超限逐步判定 |
| I9 | 确认超时 | **MVP不设服务端超时自动执行** | 会话可保持等待直到确认/取消 |
| I10 | 确认等待中身份变更 | 确认前再查须使用确认当下的 role | 用于H2样例；不得沿用过期身份放行 |

### 4.3 调试能力验证清单

| 检查项 | 通过标准 | 验证方法 |
|-------|---------|---------|
| 身份切换 | 切换 owner/guest 后，下一次编排使用新 role | 操作：UI切换身份 → 提交编排 → 检查日志 role |
| 车态修改 | 修改 speedKmh/gear/batteryPct 后，下一次编排吃到新状态 | 操作：调试页改车态 → 提交编排 → 检查再查结果变化 |
| 构造权限拒绝 | 置 guest + 授权类意图 → 该步必须 reject | 操作：guest身份 → 发「解锁车门」→ 验证 reject + 原因含权限 |
| 构造模糊确认 | 置模糊参数（如「留条缝」）→ 必须 confirm | 操作：owner身份 → 发模糊指令 → 验证 confirm 而非 execute |
| 再查触发 | confirm等待中改车态 → 原可执行步变为 blocked | 操作：发起编排 → confirm等待 → 改车态 → 点确认 → 验证 blocked |

### 4.5 安全指标检查

| 检查项 | 要求 | 检查方法 | 失败级别 |
|-------|------|---------|---------|
| 越权操作拦截率 | 100% | 安全边界测试套件（R4/R4a） | BLOCKED |
| 高风险操作确认覆盖率 | 100% | 测试用例覆盖率统计（R5） | BLOCKED |
| 违规状态操作拦截率 | 100% | 车况校验测试套件（R6） | BLOCKED |
| AI不得绕过确认 | 0 | 模糊指令直执次数（R3） | BLOCKED |
| 执行日志完整性 | 100% | 日志审查（R8） | WARNING |

### 4.6 检查结果评级规则

| 评级 | 条件 | 含义 |
|-----|------|------|
| ✅ PASS | R1-R10均可现场演示或用固定脚本复现 + 安全指标100%达标 | 可发布 |
| ⚠️ WARNING | 安全指标100%达标 + Demo含主路径+至少一次规则触发 | 有优化项，评审后决定 |
| ❌ BLOCKED | 安全指标未达100% 或 无法演示一正一反 | 存在安全风险，禁止发布 |

## 6. 附录

### 6.1 MVP 身份与权限（来自 Goals）

#### 6.1.1 身份取值（恰好 2）

| `role` | 含义 |
| --- | --- |
| `owner` | 车主（驾驶员） |
| `guest` | 临时访客 |

#### 6.1.2 MVP 权限表（最小集）

| 对象 / 动作类型 | `owner` | `guest` |
| --- | --- | --- |
| 车窗调节（含模糊开度） | 允许进入后续判定（可为 execute 或 confirm） | 允许进入后续判定 |
| 空调开关/设温 | 允许进入后续判定 | 允许进入后续判定 |
| 门锁解锁、授权他人、共享钥匙类 | confirm 或 execute | **必须 reject** |

### 6.2 MVP 车控对象（最少集）

| 对象 | 示例意图 | 典型判定 |
| --- | --- | --- |
| 车窗（主驾） | 「留条缝」「开一半」 | 模糊 → confirm/补参 |
| 空调 | 明确温度/开关 | 参数全则可 execute |
| 门锁/授权类 | 访客试图授权或解锁 | guest → reject |

### 6.3 判定与结果状态机

#### 步骤判定（规则引擎输出）

| decision | 含义 | 人是否必须动手 |
| --- | --- | --- |
| `execute` | 当前车态+权限下允许模拟执行 | 否 |
| `confirm` | 允许但有风险或参数不清，须人确认/补参 | 是 |
| `reject` | 权限或安全不允许 | 否（仅展示原因） |

#### 步骤执行结果

| result | 含义 |
| --- | --- |
| `success` | 模拟执行成功并写回相关车态 |
| `blocked` | 再查未通过，或因部分失败策略未执行 |
| `failed` | 模拟执行尝试失败（如对象故障） |
| `cancelled` | 用户取消确认 |

### 6.4 验证样例清单（产品侧最小集）

#### 6.4.1 题面 + 闭环

- C1 模糊开度（S2）
- C2 访客拒绝（S3）
- C3 主路径正例（S1）

#### 6.4.2 自补隐蔽样例（≥3，P1）

| ID | 样例 | 想打的判断点 |
| --- | --- | --- |
| H1 | 行驶中（speed>0 或 gear=D）请求危险操作 | 安全联锁 reject/confirm |
| H2 | 确认等待期间调试页把身份改为 guest 再确认 | 再查权限 → blocked/reject |
| H3 | 一句含两对象且第二步依赖第一步；第一步模拟 failed | I7 后续 blocked |
| H4（可选备份） | 超长指令 / 非法 gear | I1/I5 输入边界 |

### 6.5 体验约束（给 Design）

1. 主路径应在同一前端主流程内完成，无需跳转多系统
2. 判定三态在视觉上可区分（具体视觉语言由 Design 定）
3. 确认交互必须展示：将执行什么、为何需确认、确认/取消后果
4. 时间线按时间或步骤序完整展示，答辩时可指认 B1
5. 调试能力可与主控台同页分区或同应用另一路由，但须同原型可访问
