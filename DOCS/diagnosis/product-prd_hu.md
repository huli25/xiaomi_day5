### 1.3 用户痛点

- **多目标复杂性**：用户一句复合指令可能包含多个控制目标（如"打开车窗、关空调、调低座椅加热"）
- **参数模糊性**：用户口语表达天然存在模糊性（如"留条缝"、"凉一点"、"稍微高点"）
- **动态车况**：车辆状态（车速、挡位、电量、故障码）实时变化，安全边界随之动态调整
- **权限管控**：不同角色需要不同的操作权限，越权操作必须拦截
- **行车安全**：行驶状态下某些高风险操作必须强制确认或禁止

## 2. 目标

### 2.1 核心目标

构建任务编排与安全校验体系，实现三大核心目标：

| 核心目标 | 描述 | 关键指标 |
|---------|------|---------|
| **安全** | 越权操作零容忍，高风险操作强制确认 | 越权拦截率100%、确认覆盖率100% |
| **可控** | 多任务执行可编排、可中断、可回滚 | 冲突检测覆盖率100% |
| **可追溯** | 全链路日志记录，支持完整审计回溯 | 日志完整率100% |

### 2.2 功能目标

| 编号 | 功能模块 | 功能描述 | 优先级 |
|-----|---------|---------|--------|
| F1 | 任务编排引擎 | 复合指令结构化拆解、多任务执行调度 | P0 |
| F2 | 安全校验模块 | 权限校验、车况校验、风险等级判定 | P0 |
| F3 | 确认交互模块 | 模糊参数确认、高风险操作确认、越权拒绝 | P0 |
| F4 | 执行监控模块 | 执行状态跟踪、异常处理、日志记录 | P0 |
| F5 | 权限管控模块 | 角色权限管理、临时授权、安全边界控制 | P0 |
| F6 | 行驶场景适配 | 行驶状态下确认交互极简化，优先语音交互 | P1 |
| F7 | 应急响应策略 | 异常情况下的安全降级机制 | P1 |

## 3. 非目标

以下功能不在本项目范围内：

| 编号 | 非目标 | 说明 |
|-----|-------|------|
| N1 | 不做语音识别 | 假设语音转文本已完成，系统接收文本输入 |
| N2 | 不做自然语言生成 | 确认话术和反馈为预定义模板 |
| N3 | 不做车辆硬件控制 | 只输出控制指令，不直接操作硬件 |
| N4 | 不做机器学习模型训练 | 使用规则引擎，不涉及模型训练 |
| N5 | 不做远程OTA升级 | 规则配置为本地文件，不涉及云端同步 |
| N6 | 不做多车协同 | 单车单系统，不涉及车与车通信 |

## 6. 附录

### 6.1 角色权限分级定义

| 角色 | 权限等级 | 可执行操作 |
|------|---------|-----------|
| 车主 | 4（最高） | 全部操作，包括权限管理、驾驶模式切换 |
| 主驾驶员 | 3 | 车辆核心控制（启动、除雾、行车记录仪等） |
| 车内乘客 | 2 | 分区设备控制（分区空调、对应车窗） |
| 临时访客 | 1（最低） | 基础舒适功能（娱乐系统、氛围灯等） |

### 6.2 风险等级定义

| 风险等级 | 定义 | 确认要求 | 示例 |
|----------|------|---------|------|
| 高风险 | 可能影响行车安全或车辆资产 | 必须人工确认 | 车门解锁、驾驶模式切换 |
| 中风险 | 可能影响用户体验但不涉及安全 | 条件确认 | 车窗全开、空调最大档 |
| 低风险 | 日常舒适功能 | 无需确认直接执行 | 氛围灯调节、音乐播放 |

### 6.3 车况状态约束矩阵

| 车况状态 | 受限操作 | 限制条件 |
|----------|---------|---------|
| 行驶中（D挡） | 车门解锁、车窗全开、驾驶模式切换 | 禁止执行 |
| 低电量（<20%） | 座椅加热max、大功率设备开启 | 提示确认 |
| 充电中 | 动力系统相关操作 | 禁止执行 |
| 故障状态 | 故障关联功能 | 强制禁用 |

### 6.4 安全门禁触发条件

- 每个版本发布前必须通过安全门禁检查
- 新增高风险操作类型必须重新评估安全门禁
- 安全架构师签字确认后方可进入下一阶段
