# Design · 三套技术方案（不拍板）

> **交付链第 3 棒** · 上游：`docs/diagnosis/product-prd.md` + 题 05 题面  
> 版本：v0.5 · 2026-07-16  
> **本文只列出三套可选的技术路线（用什么技术），不写页面/模块细设计，也不在文内确认最终方案。**  
> **选择前提：三套都必须能实现题面能力**（见 §0）；取舍只比较技术成本与形态，不得选削弱题面的路线。

最终取舍 → `DOCS/options/`、`DOCS/decision/`。

## 0. 题面合规前提（三套共用，入选门槛）

无论选哪套技术，实现时都必须覆盖：

| 类别 | 必须具备 |
| --- | --- |
| 场景 | 多目标/模糊/情境；**拆解 ≠ 执行准出**；权限、安全、车态变化、部分失败影响结果 |
| 任务 | 自然语言→结构化任务；识别对象/参数/顺序/模糊/冲突；按车态·身份·规则给出执行/确认/拒绝；按序模拟执行并记录成功/失败/阻止 |
| 分两端 | **服务端**持车态、权限、安全规则、编排、日志；**前端**展示拆解、确认风险、看结果；**调试**可调车速/挡位/电量/故障 |

下文三套均按「能满足上表」来设计技术组合；差异只在栈与拆解实现手段。

## 方案一 · 前后端分离 + 规则模板拆解

### 使用技术

| 层级 | 技术 |
| --- | --- |
| 前端 | React 18、Vite、TypeScript |
| 后端 | Node.js、Express、TypeScript |
| 接口 | JSON REST（HTTP） |
| 运行时状态 | 服务端进程内内存（车态 / 会话） |
| 任务拆解 | 本地规则 / 关键词 / 正则模板（无外部模型） |
| 安全判定 | 服务端规则引擎（权限 + 车态 + 模糊/冲突 → execute/confirm/reject） |
| 执行与记录 | 服务端模拟执行 + 会话事件日志 |
| 工程形态 | 两个进程：前端 dev server + 后端 API 服务 |
| 仓库大致划分 | `SRC/frontend` · `SRC/backend` |

### 相对另两套（仅技术取向）

依赖少、离线可演示；拆解覆盖靠维护模板。

## 方案二 · Next.js 全栈同仓 + 规则模板拆解

### 使用技术

| 层级 | 技术 |
| --- | --- |
| 全栈框架 | Next.js（App Router）、TypeScript、React |
| 前端 UI | Next 页面 / Client Components（浏览器端） |
| 服务端 | Next Route Handlers（`app/api/*`）+ 服务端模块 |
| 接口 | 同源 HTTP JSON（`/api/...`） |
| 运行时状态 | 服务端单例内存（建议挂 `globalThis` 防热更新丢失） |
| 任务拆解 | 同方案一：本地规则 / 关键词 / 正则模板 |
| 安全判定 | 同方案一：服务端规则引擎 |
| 执行与记录 | 同方案一：模拟执行 + 日志（在 server 模块内） |
| 工程形态 | **单进程**；职责上仍是「浏览器前端 + 服务端 API」两端 |
| 仓库大致划分 | `SRC/web`（`app/` + `components/` + `server/`） |

### 相对另两套（仅技术取向）

一套命令启动；前后端边界靠目录约定，联调路径短。

## 方案三 · 前后端分离 + 模板优先 / LLM 补充拆解

### 使用技术

| 层级 | 技术 |
| --- | --- |
| 前端 | React 18、Vite、TypeScript（同方案一） |
| 后端 | Node.js、Express、TypeScript（同方案一） |
| 接口 | JSON REST（同方案一） |
| 运行时状态 | 服务端进程内内存（同方案一） |
| 任务拆解 | **模板优先**；未命中再调 **外部 LLM HTTP API**（OpenAI 兼容或国产网关）；**不训练自有模型** |
| 拆解护栏 | 输出 JSON Schema 约束 + sanitize（强制模糊标记、剔除不可信精确参数）；**丢弃模型给出的任何执行决策** |
| 安全判定 | 服务端规则引擎（唯一 `decision` 来源，与方案一相同语义） |
| 执行与记录 | 服务端模拟执行 + 日志 |
| 配置 | 环境变量：`LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`；无 Key/超时时**降级为仅模板**（不得跳过规则） |
| 工程形态 | 两进程（同方案一） |
| 仓库大致划分 | `SRC/frontend` · `SRC/backend`（后端增 LLM 调用与 sanitize 模块） |

### 相对另两套（仅技术取向）

口语覆盖更弹性、便于留 AI 协作证据；依赖网络与 Key，需降级策略保题面底线。

## 三方案技术对照（不裁决）

| 技术点 | 方案一 | 方案二 | 方案三 |
| --- | --- | --- | --- |
| 题面门槛 | 须满足 §0 | 须满足 §0 | 须满足 §0 |
| UI 技术 | React + Vite | React + Next | React + Vite |
| 服务端技术 | Express | Next Route Handlers | Express |
| 进程数 | 2 | 1 | 2 |
| 拆解技术 | 规则模板 | 规则模板 | 模板 + 外部 LLM |
| 判定技术 | 服务端规则引擎 | 同左 | 同左（LLM 不准出） |
| 额外依赖 | 无 | 无 | LLM API / 网络 |

页面布局、组件拆分、接口字段、规则表细则等 → 留给选定方案后的 Tasks / 实现，不在本文展开。
